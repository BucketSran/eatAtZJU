# Demo Script

日期：2026-05-08

## 1 分钟版本

1. 打开首页：一句话说明“食在浙大帮学生减少今天吃什么的决策成本”。
2. 点击“随机一餐”：展示系统能快速给出一个可解释的选择。
3. 进入“发现”：按标签或预算筛选，说明它不是泛泛点评，而是校园场景化决策。
4. 打开详情页：展示高德评分、公开来源、系统整理说明和推荐理由。
5. 打开“榜单”：展示午饭速决、30 元以内、夜宵、一人食、聚餐。

## 3 分钟版本

1. 首页：说明学生场景和公开 POI 初始数据。
2. 今日首推：解释长期推荐信号是“学生评价 80% + 公开信息 20%”；当前无学生评价时为公开 POI 冷启动。
3. 随机增强：在发现页选择预算或场景，再摇一餐。
4. 详情页：展示餐厅画像、公开信息整理和收藏；菜品/学生短评等待真实 UGC。
5. 榜单页：用榜单讲产品价值，不同场景对应不同决策方式。
6. 我的页：展示偏好配置，说明后续接 Supabase Auth 后会同步到用户资料。
7. 结尾：说明下一阶段是真实 Supabase/Vercel preview、Auth、UGC 审核和真实数据替换。

## 5 分钟版本

### 开场

“我们做的是浙大学生专属的校园美食发现平台。它不是复刻大众点评，而是把学生最常见的问题具体化：近不近、贵不贵、适不适合一个人、能不能聚餐、晚上还能吃什么。”

### 核心路径

1. 首页展示今日首推和随机一餐。
2. 发现页展示搜索、筛选、排序。
3. 使用随机增强，选择“30 以内”或“夜宵”再摇一餐。
4. 打开详情页，看高德评分、公开来源和系统整理说明。
5. 收藏餐厅，再进入收藏页验证状态。
6. 打开榜单页展示多个决策场景。

### 技术说明

- 前端：React + TypeScript + Vite。
- 后端：Vercel Serverless Functions。
- 数据：当前已接 Supabase，首批餐厅来自公开 POI 校验；学生评价和菜品不自动编造。
- 权限：Supabase RLS 已有初始迁移。
- API：前端优先读 `/api/*`，失败回退本地 seed。

### 风险边界

- 当前餐厅是公开 POI 初始数据，不是学生评价；系统整理评论只说明来源，不代表学生实地体验。
- 真实 UGC 上线前必须完成身份、审核、内容安全和隐私政策。
- service-role key 只在服务端环境变量，不能进入浏览器。

## 常见提问

**为什么不用大众点评？**  
大众点评是通用城市消费平台；这里强调校园场景、学生预算、距离和饭搭子社交。

**数据怎么保证可信？**  
真实版本会使用校园身份、审核队列、贡献信用和异常检测。当前 demo 只展示产品流程。

**为什么先 Web/PWA？**  
React + Vercel + Supabase 更适合快速迭代和 AI 协作，后续可以复用后端给小程序客户端。
