# 真实内容与审核运营闭环实验计划

**目标**：用可追溯、可回滚、尽量自动化的方式替换当前 demo seed，先产出一批“非虚构、可上线试用”的紫金港周边餐饮数据。

**核心判断**：不建议直接爬取小红书/大众点评正文、图片或用户评论。更稳的路线是用“互联网线索层”发现候选店，再用地图 POI / 官方页面 / 可公开引用链接验证事实字段，最终由系统自动生成“待审核草稿”，用规则审核 + 抽样复核替代全人工逐条审核。

## Claim Map

| Claim | Why It Matters | Minimum Convincing Evidence | Linked Blocks |
|---|---|---|---|
| C1: 可以自动构建第一批真实餐厅数据 | 解决虚拟 seed 影响可信度的问题 | 至少 80 家候选 POI，自动去重后 40-60 家可入库，字段完整率超过 90% | B1, B2, B3 |
| C2: 自动审核足够安全，不会严重污染数据 | 用户没有时间人工审核，需要先自动跑起来 | 自动通过项具备来源、坐标、人均/评分/标签证据；高风险项进入 pending，不直接公开 | B3, B4, B5 |

## 数据源策略

### 一级数据源：地图 POI API

推荐优先用高德 Web 服务 API。

可取字段：

- 餐厅名
- POI ID
- 地址
- 经纬度
- POI 类型
- 商圈/区域
- 评分
- 人均消费
- 特色 tag/菜品字段
- 距离

限制：

- 需要高德开放平台 Web 服务 Key。
- 搜索接口不是全量导出，分页最多有限制，需要按关键词、周边、多边形组合查询。
- 不直接复制第三方平台评论正文。

### 二级数据源：官方页面

用于校内食堂、后勤餐饮点。

可取字段：

- 食堂/餐饮点名称
- 所在校区/楼层/方位
- 经营品类描述
- 官方来源 URL

### 三级数据源：人工可选输入

如果用户后续愿意给小红书/大众点评链接，只作为“候选线索”，不复制正文、不保存图片、不搬运点评。系统只抽取事实字段，并生成待审核来源引用。

### 四级数据源：互联网线索层

用于发现“地图 API 泛搜不一定排在前面，但学生近期确实提到过”的候选店。

允许使用：

- Bilibili 搜索结果页/视频标题/公开视频基本信息。
- 普通搜索引擎结果摘要。
- Trip.com / 携程 / Apple Maps / Tripadvisor 等公开餐厅页的结构化事实字段。
- 学生新生指引、校友指南、论坛公开贴的店名线索。

限制：

- 只保存“线索 URL、标题、发布时间/抓取时间、疑似店名”，不复制长文案、弹幕、评论或视频内容。
- 只接受近 2 年线索，无法确认发布时间的线索降权。
- 线索必须经过高德 POI 存在性校验，校验失败不得自动发布。
- 线索来源不能作为评分或评论，只能作为“有人近期提到过”的发现信号。

## MCP / Tool 配置建议

### 必需账号与密钥

- `AMAP_WEB_SERVICE_KEY`：高德开放平台 Web 服务 API Key。
- `SUPABASE_DB_URL`：已配置，用于导入候选数据。
- `SUPABASE_SERVICE_ROLE_KEY`：已配置，用于服务端导入、审核、回滚。

### 可选 MCP

- 高德地图 MCP Server：用于 POI 搜索、地理编码、路线/距离估算。
- Browser MCP / mcp-chrome：用于打开官方页面、检查后台导入结果、抽样查看线上页面。
- GitHub / Vercel skills：用于版本管理、部署和 smoke。

### 不建议配置

- 小红书/大众点评自动化爬虫账号。
- 需要绕过登录、反爬、验证码或平台访问限制的工具。
- 自动下载用户图片或复制评论内容的工具。

## 实验 Blocks

### B1: POI 候选采集

**Claim tested**：地图 API 能产生足够真实候选。

**输入**：

- 中心点：紫金港校区附近坐标。
- 半径：1km、2km、3km 三档。
- 关键词：餐饮、美食、中餐、快餐、小吃、咖啡、甜品、面馆、烧烤、火锅、外卖。
- 类型码：餐饮服务相关类型。

**输出**：

- `data/raw/amap_poi_YYYYMMDD.json`
- 每条保留 `source_refs`，包括 API、POI ID、查询参数、采集时间。

**成功标准**：

- 原始候选不少于 80 条。
- 去重后不少于 40 条。
- 每条至少有名称、地址、坐标、来源。

### B2: 官方校内餐饮补全

**Claim tested**：官方页面能补齐校内食堂和餐饮点。

**输入**：

- 浙江大学后勤集团餐饮服务页。

**输出**：

- `data/raw/zju_food_official_YYYYMMDD.json`

**成功标准**：

- 至少提取 8-15 个校内餐饮点或食堂相关条目。
- 每条有官方来源 URL。

### B2.5: 互联网近两年线索发现

**Claim tested**：互联网公开内容能补充地图泛搜遗漏的学生热点店。

**输入**：

- 搜索关键词：
  - `浙大 紫金港 美食 2024`
  - `浙大 紫金港 餐厅 2025`
  - `site:bilibili.com 浙大 紫金港 美食`
  - `浙江大学 紫金港 附近 美食`
- 时间过滤：
  - 优先 2024-05-09 之后发布或更新的内容。
  - 搜索结果只显示抓取时间但无发布时间时，标记 `recency_status = unknown`，置信度降权。

**处理规则**：

1. 抽取候选店名，不保存正文和评论。
2. 用高德关键字搜索验证候选店是否存在。
3. 用 POI 坐标判断是否在紫金港 3km 范围内。
4. 若高德返回多个结果，使用名称相似度、距离、餐饮类型过滤。
5. 只把通过高德验证的候选进入 B3。

**输出**：

- `data/raw/web_clues_YYYYMMDD.json`
- `data/processed/web_clue_amap_validation_YYYYMMDD.json`

**成功标准**：

- 至少发现 10 个非校内食堂候选。
- 高德验证通过率 >= 60%。
- 每条候选有 `clue_source_url` 和 `amap_poi_id`。

### B3: 标准化与自动标签

**Claim tested**：候选能转成当前 `restaurants` schema。

**规则**：

- 根据距离和地址推断 `area`、`walk_minutes`、`distance`。
- 根据 POI type / tag / name 推断 `cuisine`。
- 根据人均推断 `price`。
- 根据名称/类型推断 `service_modes`、`meal_periods`、`scenario_tags`、`constraint_tags`、`preference_tags`。
- 缺少评分时使用保守默认值，并标记 `confidence < 0.65`。

**输出**：

- `data/processed/restaurant_candidates_YYYYMMDD.json`
- `data/processed/review_report_YYYYMMDD.json`

**成功标准**：

- 字段完整率超过 90%。
- 自动标签覆盖率超过 80%。

### B4: 自动审核策略

**Claim tested**：不靠人工逐条审核也能避免明显污染。

**自动通过条件**：

- 坐标在紫金港周边设定边界内。
- 餐厅名非空，地址非空。
- 来源数 >= 1。
- 与现有餐厅相似度不超过阈值，或被判定为同一餐厅并合并。
- price/rating 合理。
- 标签数量 >= 2。
- `confidence >= 0.78`。

**进入 pending 条件**：

- 疑似重复但无法合并。
- 坐标偏离。
- 名称像非餐饮。
- 人均/评分异常。
- 标签不足。
- 只有社交平台线索。

**拒绝条件**：

- 非餐饮。
- 明显不在目标范围。
- 缺少来源。
- 名称或地址严重异常。

**成功标准**：

- 自动通过比例控制在 50%-75%。
- pending 留给后续慢慢补，不阻塞上线。

### B5: 安全替换当前虚拟数据

**Claim tested**：可以替换 demo seed 且可回滚。

**方案**：

1. 真实数据先进入 `submissions` 或 staging JSON。
2. 生成批次 ID：`realdata_YYYYMMDD_HHMM`。
3. 自动通过项写入 `restaurants`，`source_refs` 记录批次和来源。
4. 旧 demo seed 不删除，先标记 `archived` 或单独备份。
5. 生成 rollback SQL。
6. 跑 smoke。
7. 部署。

**成功标准**：

- 线上列表可查。
- 详情页不报错。
- 推荐接口返回真实数据。
- 可通过批次 ID 回滚。

## Run Order

| Milestone | Goal | Runs | Decision Gate | Risk |
|---|---|---|---|---|
| M0 | 准备密钥和边界 | 配置 `AMAP_WEB_SERVICE_KEY`，确定紫金港边界 | 能成功请求 1 个 POI | Key 权限/额度不足 |
| M1 | 采集候选 | 高德周边/关键字/多边形查询 | 候选 >= 80 | 重复多、噪音多 |
| M1.5 | 互联网线索补充 | 搜索近两年 B 站/搜索引擎/公开指南，抽取店名 | 线索候选 >= 10 | 线索过旧或噪音高 |
| M2 | 高德验证与标准化 | 线索候选必须匹配高德 POI，转成内部 schema | 字段完整率 >= 90% | 标签推断不准 |
| M3 | 自动审核 | 置信度和重复检测 | 自动通过 40+ | 误通过 |
| M4 | staging 导入 | 只写 staging/submissions | smoke 通过 | 数据污染 |
| M5 | 正式替换 | 写 `restaurants` 并部署 | 线上 smoke 通过 | 推荐质量波动 |

## 自动审核评分

建议 `confidence` 计算：

- `+0.25` 有地图 POI ID。
- `+0.20` 有坐标且在边界内。
- `+0.15` 有地址。
- `+0.15` 有类型/菜系。
- `+0.10` 有评分或人均。
- `+0.10` 标签 >= 2。
- `+0.05` 与官方/第二来源交叉验证。
- `+0.05` 有近 2 年互联网线索，且高德 POI 校验通过。
- `-0.20` 疑似重复。
- `-0.25` 坐标偏离。
- `-0.20` 名称疑似非餐饮。
- `-0.15` 互联网线索无发布时间或超过 2 年。
