# 标签与筛选映射规则

## 单一数据源

餐厅标签、场景标签、同义词映射、硬筛选标签和软推荐标签统一维护在：

- `src/shared/restaurantTaxonomyData.json`

前端通过 `src/constants/restaurantTaxonomy.ts` 读取这份 JSON；Vercel API 通过 `server/api/_shared/restaurantService.cjs` 读取同一份 JSON。不要在页面、组件或 API 服务里重新手写一份标签映射。

## 标签分层

硬筛选标签会直接缩小候选餐厅范围：

- `正餐`
- `饮品`
- `食堂`
- `非食堂`
- `辣`
- `不辣`
- `清真友好`

软推荐标签只影响推荐排序和文案，不应该作为硬过滤条件：

- `近`
- `实惠`
- `一人食`
- `聚餐`
- `夜宵`
- `面食`
- `轻负担`
- `大份`
- `快乐碳水`
- `咖啡`
- `奶茶`
- `甜品`

## 发现页使用原则

发现页不再让用户面对一整墙标签，而是按“场景 -> 约束 -> 排序偏好”组织：

- 场景：快速正餐、外卖、非食堂、夜宵、饮品、聚餐、一人食。
- 约束：校区、大类、堂食/外卖、时间、预算、距离、辣度、清真、食堂/非食堂。
- 排序偏好：近、实惠、一人食、聚餐、夜宵、面食、快餐等只进入推荐分。

其他偏好应进入 `RecommendationContext.preferences` 做排序加权，而不是进入 `tags` 做硬过滤。这样可以避免用户选择“近 + 实惠 + 一人食”后结果突然清空。

## 餐段多选规则

餐段是多选维度，不是互斥标签。一个餐厅可以同时适合 `中餐`、`晚餐` 和 `夜宵`，也可以只适合 `下午茶`。

- 新提交使用 `mealPeriods: string[]` 作为主字段。
- `mealPeriod: string` 只作为旧数据和旧接口兼容字段保留。
- 推荐和筛选时，用户当前只选一个餐段；餐厅只要 `mealPeriods` 包含该餐段即可命中。
- 如果旧数据没有 `mealPeriods`，前后端应回退到 `mealPeriodTagMap`、`tags` 或 `suitedFor` 做兼容匹配。
- 批量补数据前必须先生成审计报告，不直接污染线上数据。

## 我的页使用原则

我的页只展示常用吃饭情境和少量底线，不直接展示后台信号：

- `profilePreferenceScenes`：用户能看懂的常用情境，例如赶课快吃、不想吃食堂、一人食、聚餐改善、夜宵、饮品甜点。
- 少量底线：非食堂、清真友好、辣、不辣。
- 后台信号仍保存在偏好里，用于首页、发现页和榜单排序。

## 贡献页使用原则

贡献页不要要求投稿者理解标签系统。表单只让用户填写：

- 餐饮类型：正餐 / 饮品 / 全部。
- 校区、位置、菜系、人均、评分、推荐理由。
- 可选补充：堂食/外卖、餐段多选、辣度、饮食限制、关联餐厅。

提交时根据标题、菜系、位置、备注和人均自动整理标签，并把自动整理结果写入待审核内容。管理员后续可以在审核环节修正，不把标签选择压力前置给普通用户。

## API 使用原则

- `tags`：只传硬筛选标签。
- `preferences`：传软推荐标签、用户偏好、场景、校区、饭点等排序信号。
- `tag`：保留旧接口兼容，只用于单个硬筛选或旧调用。

## 增加新标签前检查

1. 这个标签是否真的会让用户想排除餐厅？如果不是，默认放入 `softRankingTags`。
2. 是否和已有标签语义重复？优先扩展 `taxonomyTagMap` 同义词，不新增展示标签。
3. 是否需要用户直接点击？如果不是，只放入映射或 seed 数据，不放到发现页筛选按钮里。
4. 是否会和现有标签冲突？冲突项必须加入互斥组。
